# CUDA-Q QAOA API Validation Report

This report documents the validation and characterization of key CUDA-Q APIs, highlighting known issues and establishing correct usage patterns for the QAOA solver.

---

## 1. `exp_pauli` USAGE & MULTI-TERM RESOLUTION

### The Issue
The `kernel.exp_pauli` operation only accepts a `SpinOperator` consisting of a **single Pauli term** (e.g., $Z_0$ or $Z_0 Z_1$). Passing a multi-term Hamiltonian (e.g., $Z_0 + Z_1$) directly triggers:
```
RuntimeError: error: exp_pauli operation requires a SpinOperator composed of a single term.
```

### The Solution
To evolve under a multi-term cost Hamiltonian $H_C = \sum_t c_t P_t$ (where $P_t$ are Pauli terms and $c_t$ are real coefficients), we must decompose the Hamiltonian and apply `exp_pauli` term-by-term.

### Angular Scaling & Coefficient Handling
1.  **Coefficient Disappearance**: `kernel.exp_pauli` ignores the coefficient $c_t$ of the term passed to it. The generated MLIR shows a pure Pauli string rotation. Therefore, we must manually scale the rotation angle by the term's coefficient:
    $$\theta_t = 2.0 \times \gamma \times c_t$$
2.  **Angle Calibration**: We calibrated the rotation angle by measuring the expectation value of the $X$ operator on a state $|+\rangle$ after a $Z$ rotation. For an angle parameter of $\theta = 0.5$, the expectation value was exactly $\cos(2 \theta) = \cos(1) = 0.5403023$. This confirms that `exp_pauli` implements:
    $$U(\theta) = \exp\left(-i \frac{\theta}{2} H_{\text{term}}\right)$$
    To implement the QAOA evolution $\exp(-i \gamma c_t P_t)$, we must pass the parameter:
    $$\theta = 2.0 \times \gamma \times c_t$$

## 2. `SpinOperator` ITERATION

CUDA-Q represents Hamiltonians using two main classes:
1.  `cudaq.spin.SpinOperator`: Represents a general sum of Pauli terms (e.g., $Z_0 + Z_1$). Calling `list()` on a `SpinOperator` returns a list of `SpinOperatorTerm` objects.
2.  `cudaq.mlir._mlir_libs._quakeDialects.cudaq_runtime.SpinOperatorTerm`: Represents a single Pauli term (e.g., $1.5 \times Z_0 Z_1$). Calling `list()` on a `SpinOperatorTerm` returns a list of `SpinOperatorElement` objects (representing individual single-qubit Pauli gates like $Z_0$ and $Z_1$).

### Correct Iteration Pattern
To extract the terms and coefficients of any Hamiltonian for term-by-term evolution:
*   If the Hamiltonian is of type `SpinOperator`, we can iterate over it directly.
*   For each term, we must use `term.evaluate_coefficient().real` to get its real coefficient (avoiding the deprecated `term.get_coefficient()` method).
*   We can pass the term directly to `exp_pauli` as the Pauli term description, but we must scale the parameter by the coefficient.

## 3. COBYLA OPTIMIZER API

The classical optimizer `cudaq.optimizers.COBYLA` is configured and executed as follows:
```python
optimizer = cudaq.optimizers.COBYLA()
optimizer.max_iterations = max_steps
optimizer.initial_parameters = [...]  # List of 2*p floats
energy, opt_params = optimizer.optimize(2 * p, objective_function)
```
Where `objective_function` is a Python callable that takes a list of floats (parameters) and returns a float (expectation value).

---

## 4. `observe()` API

The `cudaq.observe` API computes the expectation value of a Hamiltonian on a state generated by the kernel:
```python
result = cudaq.observe(kernel, hamiltonian, params, shots_count=shots)
energy = result.expectation()
```
*   **Noiseless Statevector Simulation**: If `shots_count` is omitted, `cudaq.observe` runs a noiseless simulation, returning the exact mathematical expectation value. This is highly recommended for toy validation to ensure optimization convergence without shot noise.
*   **Noisy / Shot-based Simulation**: Specifying `shots_count=shots` runs a shot-based simulation, returning a noisy expectation value estimate.

---

## 5. `sample()` API

The `cudaq.sample` API samples bitstrings from the state vector generated by the kernel:
```python
counts = cudaq.sample(kernel, params, shots_count=shots)
```
*   **Automatic Measurement**: If the kernel does not contain explicit measurement gates (`kernel.mz`), `cudaq.sample` automatically measures all allocated qubits at the end of the circuit execution.
*   **Parameters**: Accepts kernel parameters as arguments.

---

## 6. BITSTRING ORDERING

CUDA-Q uses **left-to-right (MSB-on-left)** bitstring ordering:
*   Index 0 of the returned bitstring corresponds to qubit 0 (`q[0]`).
*   Index $i$ corresponds to qubit $i$ (`q[i]`).

### Example
For a 2-qubit system where `q[1]` is flipped to $|1\rangle$ and `q[0]` remains $|0\rangle$, the sampled bitstring is `"01"`. 

This simplifies decoding because the $idx$-th qubit state matches `bits[idx]` directly, without requiring string reversal.
